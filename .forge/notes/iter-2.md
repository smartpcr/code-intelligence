## Decisions made this iter

- **Module path fix in go.mod is the structural answer to evaluator item #4.** A previous iter on a sibling branch had ALSO fixed this (CHANGELOG line 1010-1021 references it); commit `f28ab4f` re-broke it via merge. Iter 1's local `replace` workaround only proved the iter-1 code was sound -- it didn't actually fix the worktree test-runnability. This iter fixes go.mod itself so `CGO_ENABLED=0 go test ./internal/repo_indexer/...` PASSES.
- **PG-backed writer uses `pg_advisory_xact_lock(0x43435249, hash32(repo_id))` for per-repo serialisation** instead of relying on a unique partial index on `repo_event(repo_id) WHERE kind='registered'`. The 0001 migration does NOT carry such an index; rather than mutate the migration here I encode the exactly-one invariant in the writer's tx. Single-row contention is per-repo, not service-wide, so latency cost is bounded.
- **`NULLIF($3, '')` in the commit INSERT** translates application-layer empty-string `parent_sha` to DB NULL (the column is NULLABLE for first commit).
- **Separate `EnvEnableScaffoldIndexerWebhook` flag** rather than reusing `EnvEnableScaffoldChurnWebhook`. Justification: operators may legitimately want to mount the churn webhook without the indexer (or vice versa) -- they have different blast radii. The HMAC secret IS shared (architecture Sec 8.5 "one external-ingest secret"). Validation interlock requires BOTH the flag AND the HMAC secret to enable.
- **Rescan handler at `/v1/indexer/rescan` is HMAC-EXEMPT** in scaffold mode -- operators authenticate at the CLI surface (Bearer + audience, in a future stage). This matches the architecture's split between "external webhook" (HMAC) and "internal CLI rescan" (operator credential). Documented in iter-notes as an Open Question if the evaluator wants both surfaces HMAC-gated.
- **Indexer constructor block sits AFTER `buildPolicyWriter`** so the same `*sql.DB` handle is reused. Constructing it earlier would either duplicate the open or hand the indexer a nil DB.
